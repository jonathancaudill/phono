# P2.1 Session transport reconnect spike

## Finding (go/no-go)

librespot 0.8.0 `SessionInternal` uses `OnceLock` for connection dispatch, mercury, channel,
login5, spclient, etc. `Session::shutdown()` sets `invalid = true` and shuts down mercury/channel
permanently — the session object cannot be reused for in-place AP reconnect like librespot-java.

## Decision

**No-go** for transparent `Session.reconnect_transport()` without replacing all nine `OnceLock`
subsystems with resettable handles. phono uses **seamless rebuild** instead:

- `pending_queue` + `snapshot_resume()` preserve queue/position/play-state across rebuild
- `ensure_playback_ready()` single-flight rebuild from cached credentials
- `build_active()` logs rebuild reason; banked tracks hit `AudioFile::Cached` on reload
- `force_reconnect_check()` debounced 5s before `session.shutdown()`

## Future work

If handoff gap remains unacceptable on device (T1), refactor `SessionInternal` connection path
only (not login5/spclient OnceLocks) and add `transport_lost()` vs `invalidate()` split.
